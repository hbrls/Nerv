## §5. 已锁定结论（不再讨论）

### 5.1 项目目标

- **结论**：注入式读取机制本身就是项目目标——在私有化部署的 Gitlab 上，通过浏览器已登录态（person token 或 cookie），用注入式 JS（主要指 Tampermonkey）调用 Gitlab 接口读取数据。读取 issues 是验证该机制可行性的首个用例，未来机制成立后可扩展。
- **来源**：USE 4.1.1 回答。

### 5.2 读取用途

- **结论**：读取并规范化为 JSON，能在浏览器 console 打印。backstage 平台作为工作机制会消费这个 JSON，但 backstage 平台本身不在此讨论。当前目标止于"读取 + 规范化输出 JSON"。
- **来源**：USE 4.1.2 回答。

### 5.3 读取方式

- **结论**：通过内在机制（如接口）读取，不是读当前页面已加载出的 issues。历史回溯不是项目目标，但能用"能读到历史 issues"证明机制成立。
- **来源**：USE 4.1.4 回答。

### 5.4 只读约束

- **结论**：当前阶段只读取 Gitlab 数据，不涉及任何写回操作（创建 MR、评论、修改 issue 等）。
- **来源**：USE 4.2.4 回答。

### 5.5 验收标准

- **结论**：人工验收。能 console.log 出 issues 数据且有值，即可提交验收，最终人工确认。字段使用 Gitlab issues 标准字段，不自定义。
- **来源**：USE 4.3.1 / 4.3.2 回答。

### 5.6 性能与鉴权

- **结论**：人工低频使用，完全不需要考虑性能问题。鉴权方式（person token / cookie）是调研和开发目标本身，无法预先回答，是项目产出而非前置输入。
- **来源**：USE 4.3.4 / 4.3.6 回答。
