## 2. 锁定结论

### 2.1 问题 1：「读取」对象

**锁定结论**：Gitlab 数据源层接入，归属 backstage.js 数据源扩展范畴；具体数据子集（blames / 仓库源码 / MR / Issue / Pipeline 等）未在本轮锁定。

**候选来源**：

- `Index.md` 标注本轮性质为"能力 / 数据源问题"
- `Index.md` 原始信号为"优化 Gitlab 的读取"
- `C-001 §5` 假设：'Gitlab 读取'对应 backstage.js 的数据源扩展，而非 backstage 之外的能力

**假设**：

- VISION-002（重启 backstage.js 项目）已完成且产物稳定，因此 Gitlab 接入入口在 backstage.js 数据源层
- "优化"指扩展当前数据源，而非独立新建一条平行链路

### 2.2 问题 2：下游消费者

**锁定结论**：backstage.js 内部能力链路（候选：catalog 同步 / techdocs / search）；具体接入哪个能力未在本轮锁定。

**候选来源**：

- `Index.md` 原始信号"重启 backstage.js 项目，优化 Gitlab 的读取"将两者并列
- `C-001 §4` 列出下游候选：backstage.js 的某个能力（如 catalog 同步、techdocs、search）

**假设**：

- 不直接展示给人看（不是 UI 展示用途），而是喂给 backstage.js 内部能力
- "Gitlab 读取"对应 backstage.js 数据源扩展，而非 backstage 之外的能力（继承 §2.1 的同一假设）

### 2.3 问题 3：可验证的成功标准

**锁定结论**：必须同时满足以下三条才算成功（AND 关系）——

1. 能从 Gitlab 拉到数据（接口调用成功 + 拿到非空结果）
2. 拉到的数据能被既定的下游能力消费（端到端贯通）
3. 在性能 / 限频 / 鉴权约束下能稳定运行（不超出 Gitlab API rate limit、token 有效、网络可达）

**候选来源**：

- `C-001 §4` 列出成功标准候选：(a) 能拉到数据；(b) 拉到后能用于既定的下游能力；(c) 性能 / 限频 / 鉴权上能扛得住

**假设**：

- "可验证"意味着每条标准都有可执行的检查方式（手工或自动化均可）
- 三条是 AND 关系，缺一不可；本轮未给出每条的具体阈值
